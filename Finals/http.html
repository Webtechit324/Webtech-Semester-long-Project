Hypertext Transfer Protocol

- application layer used primarily to retrieve hypertext (on hypermedia) documents and reources on the World Wide Web.

- jointly developed by the W3C and the IETF.

Version History

HTTP 0.9 (1991)

- Phases: Connection -> Request -> Response -> Disconnection (4 steps)

HTTP 1.0

- Request For Comments (RFC) - 1945, May 1996

HTTP 1.1

- RFC 2068 - Jan 1997

- RFC 2616 - Jun 1999

- RFC 7230-7235 - Jun 2014

HTTP 2.0

- RFC 7540 - May 2015

Hyper Text Transfer Protocol (HTTP) Unsecure

Based on a client-server architecture.

Client, aka User Agents (UA)

- Web Browser, Web Crawlers/Spiders, Other End USer Tools and Applications

Server

- Origin Server, Proxy Server (ex. Authentication Server), Gateways, Tunnels

HTTP Resource Addressing

Identifies using Uniform Resource Identifer (URI; RFC 3986) or more specifically, HTTP URIs.

Two Subtypes of URI:

- Uniform Resource Name (URN) - Example: International Standard Book Number (ISBN), Universal Product Code (UPC)

- Uniform Resource Locator (URL) - Example: google.com

Authority

- User information/authentication credentials (deprecated)

- Host - domain name (resolved to an ip address using dns) of the server where the resource resides (or will be created).

- Port Number

Path to Resource (resolved relative to the document root on the server)

- Fragment Identifier - starts with a hashtag (#).

Protocol

- a set of rules that needs to be followed when messages are communicated in the Internet.

Examples:

- HTTP (Hypertext Transfer Protocol) - a protocol used for transferring messages over the Internet.

- HTTPs (Secured Hypertext Transfer Protocol) - a protocol with a combination of HTTP and SSL or Secure Sockets Layer which makes the sharing of information more secure.

HTTP Response Message

Status Line

- The first line of the response message. It start with the protocol version and the numeric status code with the description.

Response Header

- This allows the server to pass additional messages that cannot be placed in the Status Line.

Message Body

- The message body is used to carry the entity-body associated with the request or response. It only differs from the entity-body when transfer coding has been applied.

HTTP Request Message

- request line (carriage return line feed) - terminated line consisting)

Methods

- Standard Method

    - safe methods: get(retrieve), head, options, trace, connect

    - modify server/not safe: post, put, delete

    - idempotent: get, head, put, delete, options, trace

- Extension Method

    - WebDAV (RFC 4918) - PROPFIND, PROPPATCH, MKCOL (make a collection/create folder), COPY, MOVE, LOCK, UNLOCK

    - Options - query what the server can do

    - Trace - request a loop-back of the request message; typically used for testing/diagnostic of the request/response chain; to know where you committed error (Example: 34 - length of the section (hex 1 byte) Trace/index.html HTTP/1.1 Host: www.webtek.org (actual data))

    - Connect - establishment of a tunnel to the destination origin servers and if successful, thereafter restrict its behavior to blind-forwarding of pockets in both directions, until the tunnel is closed.

    - cacheable method

    - safe method

    - idempotent method

Request URI

- RFC 2616 - 4236 Update Required

HTTP Request Methods

GET

- Request data from a specified source.

HEAD

- Same as GET but returns only HTTP headers and no document body.

POST

- Submit data to be processed to a specified resources.

OPTIONS

- OPTIONS method represents a request for information about the communication options available on the request/response chain identified by the Request-URI.

TRACE

- TRACE method is used to invoke a remote, application-layer loop- back of the request message.

PUT

- Uploads a representation of the specified URI.

DELETE

- Deletes the specified resource.

CONNECT

- Converts the request connection to a transparent TCP/IP tunnel

Safe Methods:

- Get, Head, Options, Trace

Not Safe Methods:

- Post, Put, Delete

Idempotent:

Get, Head, Put, Delete, Options, Trace

HTTP Message Headers

General Header Fields

This header fields have general applicability for both request and response messages.

- Connection

- Data

- Pragma

- Transfer encoding

- Trailer

- Upgrade

- Via

- Warning

Request Header Fields

The request-header fields allow the client to pass additional information about the request, and about the client itself, to the server. These fields act as request modifiers.

- Accept language

- Accept encoding

- Authorization

- Proxy

- Expect

- From

- Host

Response Header Fields

The response-header fields allow the server to pass additional information about the response which cannot be placed in the Status- Line. These header fields give information about the server and about further access to the resource identified by the Request-URI.

- Accept - range

- Age

- ETag

- Location

- Proxy - authenticate

- WWW - authenticate

- Retry - after

- Server

- Vary

Entity Header Fields

This header fields define meta information about the entity-body or, if no body is present, about the resource identified by the request.

- Allow

- Content - encoding

- Content - language

- Content - length

- Content - location

- Content - MDS

- Content - range

- Content - type

- Expires

- Last - modified

Conditional Header

- Request - if-non-match

- Condition - if-modified.since

- If - ranges

- Max - forward

- Trailer encoding

- User - agent

Extended Headers

- Upgrade

- Do not track (DNT)
